Python · 自動化 · 自學

從 VBA 到全天候自動化系統：一個營運專員的 Python 自學路徑

不是因為想學程式才學程式，是因為工具不夠用才被逼出來的。這是一段從 Excel 書籤到 asyncio 併發排程的真實演進，以及每一步背後的痛點。

Chris Chou · 客服營運流程專員 @ Shopee Taiwan · 約 12 分鐘閱讀

一切從一份殘缺的報表開始

剛進 Shopee 的時候，我的程式工具箱裡只有一樣東西：Excel VBA。對大多數營運分析工作來說，這已經夠用。

直到第一個分析專案。

任務是分析某類客服案件的根本原因，找出可以改善的環節。但當我把系統報表匯出來，才發現欄位殘缺——有些關鍵資訊根本不在報表裡，必須進到系統一筆一筆查。幾千筆資料，靠人工查不現實。

我的第一個念頭是：VBA 可以控制瀏覽器，讓它自動去查。

這個念頭是對的，但執行起來很痛苦。VBA 操控瀏覽器的方式原始、不穩定，速度慢，而且一旦網頁版面有任何變動，整個腳本就壞掉。跑到一半當掉是常態，不是例外。

工具不是不能用，但你會花 80% 的時間在跟工具搏鬥，只有 20% 的時間在解決真正的問題。

就是在這個時候，我開始認真看待 Python。

工具演進的六個階段

01
Excel VBA

起點：能動，但太脆弱

VBA 控制瀏覽器的方式依賴 COM 介面，速度慢、穩定性差。網頁稍有更新就失效，維護成本極高。它讓我知道「自動化取資料」這件事是可行的，但需要更好的工具。

02
Python + Selenium

第一次升級：真正的瀏覽器自動化

Selenium 讓 Python 可以控制 Chrome，模擬真實的使用者操作。但剛開始有兩個大坑：第一是 WebDriver 設定，版本對不上、路徑設錯，光是讓瀏覽器能被控制就花了不少時間。第二是 XPath——定位網頁元素的語法，寫起來冗長，一個複雜的選擇器可能就是一行很長的字串。

解法是把重複用到的操作包成通用函式——點擊、輸入、等待元素出現、錯誤重試——讓後續寫新腳本的速度大幅提升。

03
Selenium + BeautifulSoup

過渡階段：繞過渲染，加快速度

Selenium 最大的瓶頸是需要等網頁完整載入，包含所有圖片、CSS、JavaScript 執行完畢，才能取到資料。有時候資料其實已經在 HTML 裡了，但瀏覽器還在渲染其他東西。

解法是讓 Selenium 只負責打開特定的 API 網址（通常是回傳 JSON 或 HTML 的端點），再用 BeautifulSoup 直接解析回傳的內容。不需要等完整網頁載入，速度快很多，穩定性也更高。

04
requests + API

頓悟時刻：跳過瀏覽器，直接取資料

長時間在 Chrome DevTools 的 Network 頁籤裡翻找，慢慢開始看懂系統在每個操作背後實際發出的 API 請求——URL、參數、Header、認證方式。

某天突然想到：既然我知道系統在打哪個 API，我為什麼要模擬人的操作？我可以直接打。

用 requests 直接呼叫 API，瀏覽器完全不需要打開。速度從「秒」降到「毫秒」，穩定性大幅提升，程式碼也更簡潔。

# 舊方式：Selenium 等網頁載入，再抓資料 driver.get(target_url) WebDriverWait(driver, 10).until(EC.presence_of_element_located(...)) data = driver.find_element(By.XPATH, '//div[@class="result"]').text # 新方式：直接打 API，跳過瀏覽器 response = requests.get( api_endpoint, headers={'Authorization': token}, params=query_params ) data = response.json()
05
asyncio 併發

速度瓶頸：一萬筆資料，跑一天

即使用了 requests，面對大量資料時還是遇到瓶頸。分析用的資料動輒數萬筆，若一筆一筆依序處理，假設每次 API 呼叫需要 1 秒，一萬筆就是將近 3 小時。中途網路抖一下、token 過期、任何小問題都會讓整個流程中斷，必須從頭重跑。

非技術讀者版

threading vs asyncio 的差別

threading 像是開很多個收銀台，每個台各自服務一位客人，同時進行。

asyncio 像是一個收銀員，在等客人刷卡的空檔，先去幫下一位客人掃條碼。沒有開更多收銀台，但把等待的時間利用起來了。

API 請求大部分的時間都在「等伺服器回應」，這正是 asyncio 最有效率的場景——在等 A 回應的時候，同時發出 B、C、D 的請求。

改用 asyncio 之後，同樣的一萬筆資料，時間從數小時壓縮到幾分鐘。更重要的是，流程穩定了——不再因為跑太久而中途斷掉。

06
排程 + 全天候運行

從工具到系統：讓它自己跑

單次執行的腳本，還是需要人去觸發。加入排程之後，資料更新、報表產出、異常通知，全部變成自動發生的事。每天早上打開電腦，昨天的數據已經在那裡了。

這個轉變讓「自動化」從「省去重複操作」升級成「讓系統替你持續工作」。人的注意力可以放在判斷和決策，而不是等資料跑出來。

這段路教會我的四件事

1
最好的學習動機是真實的痛苦

我不是因為「Python 很有用」才去學，是因為 VBA 讓我每天在跟工具搏鬥才去學。帶著真實問題去學，每一個新概念都有立即可以驗證的場景，學得快，也記得住。

2
工具的演進不是跳躍，是逼出來的

VBA → Selenium → BeautifulSoup → requests → asyncio，每一步都是因為前一個工具碰到了它解決不了的問題。不需要一開始就學最好的工具，需要的是在遇到瓶頸時知道往哪裡找答案。

3
DevTools 是最被低估的學習工具

花時間看 Network 頁籤，理解系統在背後做什麼，是從「模擬操作」跳到「直接打 API」的關鍵。這個習慣讓我對系統的理解從「介面層」深入到「資料層」，思考問題的方式也跟著改變。

4
把重複的東西包成函式，是從「寫程式」到「工程思維」的分水嶺

Selenium 階段把 XPath 操作包成通用函式，不只是讓程式碼更短，而是開始用「可重用、可維護」的角度思考問題。這個習慣後來在每個專案都省了大量時間。

從函式到類別：建立自己的工具庫

函式解決的是單一操作的重用，但當專案越來越多、需求越來越複雜，你會發現需要的不只是函式，而是一套可以組合、可以擴展的架構。

這個階段我開始把相關的函式整理成類別。最核心的是 DMS API 類，把所有跟系統溝通的細節封裝起來：登入流程、Session 保存與刷新、API 呼叫格式、asyncio 併發控制。另外還有各種報表生成類，把不同格式的輸出邏輯統一管理。

# 有了類別之後，新需求的程式碼變成這樣 dms = DMSClient() await dms.login() # 併發取得一萬筆案件資料 data = await dms.fetch_batch(case_ids, concurrency=20) # 直接生成報表 report = ReportBuilder(data) report.export_excel("output.xlsx")

幾行就解決，因為複雜的部分已經在類別裡了。登入怎麼處理、token 過期怎麼刷新、併發數量怎麼控制——這些細節只需要解決一次，之後每個新需求都站在這個基礎上。

工具庫是會複利的東西。每解決一個新問題，下一個問題就更快解決。時間久了，你的起點比別人高很多。

這套架構現在支撐著每日 20–30 項自動化任務，涵蓋數據更新、報表生成、異常通知等各種應用場景。面對新需求時，大部分的時間花在「想清楚要做什麼」，而不是「從零開始寫基礎設施」。

給同樣在這條路上的人

如果你現在正在用 Excel 做一些感覺「應該可以自動化」的事，或是已經在學 Python 但不知道從哪裡找真實的應用場景——我的建議是：從你現在最痛的那個重複動作開始。

不需要學完所有東西再動手。帶著問題去找答案，比照著課程學完再找問題，有效很多。工具會在你需要它的時候出現，就像 asyncio 在我為萬筆資料發愁的時候出現一樣。

這篇是我自動化學習路徑的整理，如果你也在用 Python 解決工作問題，或是有類似的演進過程，歡迎留言分享。另外一篇關於如何用 Chrome Extension 解決客服系統痛點的文章也可以參考。
