每週一小時，消失的不只是時間

CSAT（客戶滿意度）是我們團隊 OKR 裡的核心指標之一。每週都有新資料，每週都需要整理、比對、清理，然後才能分析。

在自動化之前，這個流程是手動的：從系統匯出 CSV，人工比對案件清單，標記買賣家評語、負責專員、案件類型……整個過程至少需要一小時。

一小時聽起來不多。但問題不只是時間——當你花了一小時整理資料，你往往沒有精力再去深入分析它。 最後的結果是：有資料，但沒有洞察。每週看的是當週的數字，而不是趨勢。

為什麼不用 CSV？

最直覺的自動化方式是：Python 取資料，存成 CSV，Excel 開來分析。很多人這樣做，我一開始也這樣想。但 CSV 有幾個讓我不安的特性：

格式脆弱。 有人不小心在 Excel 裡存檔，欄位可能錯位、編碼可能跑掉、日期格式可能被自動轉換。這些問題不一定馬上發現，但會讓之後的分析結果悄悄出錯。

累積困難。 CSAT 是每週持續更新的資料。CSV 要累積歷史，要麼每次手動合併，要麼覆蓋舊檔案失去歷史紀錄。

原始資料沒有保護。 分析的人直接開 CSV 操作，一個不小心存檔，原始資料就被改動了，而且不可逆。

選 SQLite 不是因為它比 CSV 酷，是因為我預判了「有人會不小心搞壞資料」這個風險，然後用架構去防止它發生。

最終的架構是：SQLite 存原始資料，Excel 透過 ODBC 連接做分析。 分析層和資料層完全分離——不管 Excel 裡怎麼操作，原始資料永遠不會被動到。

Pipeline 的三個階段

Extract

從內部系統取資料

透過 API 取出 CSAT 原始資料。因應多請求的高延遲特性，採用 asyncio 併發處理突破 I/O 瓶頸，大幅縮短等待時間。

Transform

比對、清洗、打 Tag

與案件清單比對，標記負責專員、Flow 組別、是否為 Mass 處理案件等維度。清洗資料格式，確保一致性。

Load

存回 SQLite

採用 UPSERT 邏輯寫入 SQLite 避免重複覆蓋，並啟用 WAL 模式防止讀寫鎖定衝突。Excel 透過 ODBC 安全查詢，不接觸原始資料。

# 簡化示意：整條 pipeline 的結構
async def run_pipeline():
    # Extract
    raw_data = await dms.fetch_csat(week=current_week)

    # Transform
    case_map = await dms.fetch_case_list()
    cleaned  = transform(raw_data, case_map)
    tagged   = apply_tags(cleaned)

    # Load
    db.insert(tagged)
    print(f"完成：{len(tagged)} 筆資料更新/寫入")

肉眼看不出來的 Bug

⚠ 實際踩過的坑

ReturnID 前置單引號問題

Pipeline 寫好之後，比對結果怎麼看都不對——明明應該能對到的案件，就是對不上。資料看起來沒問題，CSV 打開來看也正常，但就是比對失敗。

追查之後才發現：系統匯出的 CSV 裡，某些 ID 欄位前面有一個隱藏的單引號（'12345678），甚至尾巴常帶有不可見的空白。這些在 CSV 裡肉眼完全看不出來，但程式讀進來之後，字串比對絕對會失敗。

在資料工程中，ID 欄位並不做數學運算，所以最好的解法不是轉成整數，而是寫一個專門的清理函式，確保字串純淨並正確處理空值：

# 專門的清理函式：轉字串、去引號與空白，並防範 nan 陷阱
def clean_return_id(series: pd.Series) -> pd.Series:
    cleaned = series.astype(str).str.lstrip("'").str.strip()
    return cleaned.replace({"nan": None, "": None})
df['ReturnID'] = clean_return_id(df['ReturnID'])

這個 bug 讓我學到一件事：資料「看起來正確」和「程式讀到的正確」是兩件不同的事。 自動化之後，驗證輸出是否正確，跟寫程式本身一樣重要。

會長大的 Schema

這條 pipeline 的 tag 維度不是一開始就設計完整的，而是跟著分析需求慢慢長出來的。

欄位說明類型
case_id案件編號核心
csat_score買家滿意度評分核心
buyer_comment買家評語核心
seller_comment賣家評語核心
execution處理結果核心
agent_id負責專員Tag
flow_typeFlow 組別（A / B / C…）Tag
is_mass是否為 Mass 處理案件Tag
⋯隨分析需求持續新增Tag

核心欄位一開始就定好，tag 維度跟著需求長。每次有新的分析方向，就在 Transform 階段加一個新的標記邏輯，不需要重新設計整個 pipeline。這讓系統能持續進化，而不是每次需求改變就要重建。

Pipeline 真正發現了什麼

自動化之後，最大的改變不是節省了多少時間，而是終於有精力去看趨勢，而不只是看當週數字。

某段時間，我注意到不滿意案件數量有緩慢上升的趨勢。如果只看當週，這個變化不明顯；但跨週比對，就能看出方向。

1

發現趨勢 — 不滿意案件數連續幾週緩升，幅度不大但方向一致

2

縮小範圍 — 用 execution、flow_type 過濾，發現集中在特定 flow_type 與案件類型

3

找到根因 — 追查後發現是 PH BPO 在處理某類案件時理解有誤，導致處理方式不正確

4

針對改善 — 有了具體根因，才能針對性調整 BPO 的培訓內容和 SOP 說明

這個發現，在人工整理的時代幾乎不可能出現——不是因為資料不在，而是因為沒有時間和工具去看趨勢。

這條 Pipeline 教會我的事

1
架構選擇是風險管理，不只是技術偏好

選 SQLite 而不是 CSV，不是因為技術上更複雜，而是因為它能防止一類特定的人為錯誤。好的架構讓壞事更難發生。

2
資料「看起來正確」和「程式讀到的正確」是兩件事

單引號 bug 讓我學到：肉眼驗證是不夠的。自動化流程需要程式層面的驗證，才能確保輸出真的是你以為的那樣。

3
省下來的時間，價值不在時間本身

每週省一小時，不只是省了一小時。真正的價值是把那一小時用來做原本做不到的事——看趨勢、找根因、產生洞察。自動化改變的是你能做的事的上限。

4
Schema 不需要一開始就完美，需要能長大

從核心欄位出發，讓 tag 跟著真實的分析需求長，反而更實用、更好維護。過度設計反而是負擔。

5
讓 Pipeline 具備「冪等性」（Idempotency）

自動化最怕中途出錯重新執行時，資料重複寫入兩次。設計好 UPSERT (有則更新、無則新增) 的邏輯，確保腳本不管不小心按到幾次，資料庫的結果都永遠正確。

這是我在 Shopee Taiwan 建立的內部資料管道，用來追蹤和分析 CSAT 客戶滿意度指標。如果你也在用 Python 處理營運資料，或是對 SQLite + ODBC 這個架構有問題，歡迎留言討論。
