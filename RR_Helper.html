工具開發 · 客服營運

我如何用一個瀏覽器插件，讓 56 位客服專員每天少犯錯

從一個書籤工具開始，歷經 14 個月、45 個版本的迭代，RR Helper 是一個在大公司資源限制下，用最小成本解決前線真實痛點的故事。

Chris Chou · 客服營運流程專員 @ Shopee Taiwan · 約 10 分鐘閱讀

問題的起點

在電商客服的工作裡，有一類案件特別容易出錯：不常出現、但處理規則複雜的邊緣案例。

專員每天處理大量案件，對常見流程很熟悉，但偶爾出現的特殊案件——例如跨境物流的特殊退款規則、分期付款訂單的不同處理邏輯、特定賣家需要額外確認的步驟——因為頻率低，很容易在腦子裡慢慢淡化。等到真的碰到，就可能做出錯誤的判斷。

除了記憶問題，還有兩個明顯的效率瓶頸：

第一，系統查詢繁瑣。有些資訊分散在不同系統，專員需要打開好幾個頁面、手動計算，才能得出一個本來應該自動顯示的答案。第二，填單重複操作。填寫表單時需要從案件系統複製資訊，一個欄位一個欄位切換，每天重複幾十次，耗時又容易出錯。

這些問題不夠大，大到公司系統部門願意排進開發排程。但對前線的人來說，每天都在摩擦。

在 Shopee 這樣的大公司，提出系統優化需求走正常流程，如果效益無法量化到一定規模，通常很難排進開發優先級。這是現實，不是抱怨。

所以我決定自己解決。

第一個版本：書籤工具（Bookmarklet）

最初的解法很土，但有效。我把每一個小工具做成 Bookmarklet——把 JavaScript 塞進瀏覽器書籤，點一下就執行。

想要自動計算某個欄位？寫一段 JS，存成書籤，通知同事點一下。想要自動填表？同樣的方式。幾個禮拜內，書籤列多了七八個工具，每個解決一個痛點。

但 Bookmarklet 有一個致命問題：無法統一管理版本。

每次規則改了、流程更新了，我需要把新的書籤連結通知所有人，請他們手動替換。有些人更新了，有些人忘了，有些人手上可能一直跑著半年前的舊版。版本分歧的結果，就是同樣的案件，不同的人用不同的邏輯在處理。

這比沒有工具還要危險。

· · ·

決定做成 Chrome Extension

Bookmarklet 的問題讓我開始認真思考一個更系統性的解法。Chrome Extension 有幾個關鍵優勢：

版本集中管理。 我更新一次，所有安裝的人自動拿到新版，不需要任何人工操作。

可以串接 Firebase 做即時內容更新。 某些提醒內容（例如特殊賣家名單）需要頻繁異動，透過 Firebase 可以不用每次發新版就能更新資料。

可以在背景執行邏輯。 不只是點了才跑，可以在專員打開案件的瞬間自動判斷，彈出對應的提醒。

決定做，但我從來沒寫過 Chrome Extension。

開發過程中的三個硬骨頭

挑戰 01

從零開始學 Extension 開發，以及送審的漫長試錯

Chrome Extension 有一套自己的架構概念：Content Script、Background Service Worker、Manifest 設定……這些跟一般網頁開發不同。我花了不少時間在文件和 Stack Overflow 之間來回，更花了意外多的時間在 Chrome Web Store 的送審流程上——格式不對、描述不符規定、權限申請方式有問題，退件了幾次才順利上架。

挑戰 02

跨域查詢：在 A 網站取得 B 網站的資料

RR Helper 需要在案件管理系統（DMS）的畫面上，即時查詢其他內部系統的資料。但瀏覽器的安全機制不允許在一個網域的頁面直接存取另一個網域的 API，這就是所謂的 CORS 限制。

解法是利用 Background Service Worker 作為中間層。Content Script 把請求丟給 Background，由 Background 去向其他系統取資料，再把結果回傳給前端顯示。這個架構讓跨域查詢成為可能，也讓後來加入的多個查詢功能有了統一的架構基礎。

挑戰 03

前端 UI：我不擅長，但我知道它很重要

我的背景偏向後端邏輯和數據，前端排版是弱項。但我很清楚一件事：介面醜、操作不直覺，就沒有人願意用。 工具做得再好，使用率是零，等於沒做。

所以我在 UI 上投入了比預期更多的時間——彈窗的位置、資訊的層次、提醒的視覺權重、不同功能的入口安排。每一個細節都在想：專員在壓力下操作案件，這個介面會不會讓他更快找到他需要的東西？

從 V1.0 到現在：迭代的節奏

2025.02

V1.0 上線：三項基本提醒

最初版本只有日常提醒、新功能通知，以及組別自動判斷。功能少，但建立了核心架構。

2025.04–06

快速擴張期：功能從 3 項增至 15+ 項

加入 Remark 彙整工具、退貨商品比對、表單欄位複製工具、圖片標記工具，以及各組別的特殊提醒邏輯。也在這個階段遇到第一次線上事故：V25.05.09 因為測試不夠充分，Remark 工具出現變數設定錯誤無法正確執行，我在發現問題後立即在 V25.05.11 退回前一個穩定版本，隔天修好後重新發布。快速退版比硬撐一個壞掉的版本好，這是那次學到最重要的事。

2025.10–12

架構成熟期：跨域查詢與即時功能

加入 Re-inbound 狀態查詢、環保無包裝 TN 清單、3PL Compensation 灰度提醒。這個階段的功能開始需要跨域查詢架構的支撐，也是技術複雜度最高的一段時間。

2026.01–04

持續優化與跨地區擴展

加入動態通知功能、Buybox 商品偵測、Queue Case Export，並為 PH BPO 開發專屬功能權限，讓工具從台灣本地延伸到跨國支援。

三件這個過程教會我的事

1
工具的生命力來自解決真實的問題，不是技術的精緻程度

RR Helper 不是技術上最複雜的東西，但它解決了專員每天真實遇到的摩擦。這讓它在 14 個月後還在被使用，還在持續收到需求，而不是做完就被遺忘。

2
需求管理和版本記錄，是工具能長期維護的基礎

一開始就建立需求表流程和 Changelog 習慣，讓 45 個版本的迭代有跡可循。任何人想知道某個功能什麼時候加進來、為什麼加，都找得到答案。這不是多餘的，是工具能活下去的原因之一。

3
在大公司裡，最快的改變路徑往往是繞過審批，先做出來

等待系統部門排進開發，可能要等一年。用瀏覽器插件在兩週內解決，然後讓數字說話。這不是在批評組織流程，而是在選擇對使用者最有效率的路徑。

如果你也在大公司的前線，有類似「系統不支援但又很需要」的痛點，歡迎留言討論。有時候最好的解法不在 roadmap 裡，在你自己的書籤列。
